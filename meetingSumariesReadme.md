# Meeting Transcription + Summarization (Local) — Current State

## Goal

Provide a pipeline where:

1. A meeting recording is transcribed (with diarization) into:
   - a **flattened transcript** stored on `Meeting.transcript`
   - structured **segments** stored in `MeetingSegment` (with timestamps + speaker indices)
   - speaker labels stored in `MeetingSpeaker`

2. A meeting summary is generated by calling a local Python FastAPI service:
   - today: **stub summary**
   - plus: **Stage A extractive digest** for long transcripts (testable now)
   - later: replace stub with real model inference (AMI fine-tune)

Everything is currently designed so the **boundary contracts stay stable**.

---

## High-level Architecture

### Components

- **Frontend (browser)** calls Next API routes via `meetings.api.ts`.
- **Next.js API Routes** enforce auth and call services.
- **Services (TS)** implement business rules and call repos/clients.
- **Repos (TS)** are the only place Prisma is used.
- **Python FastAPI “python-summarizer”** returns a summary (stub now).

### Summary flow (today)

```
UI
  -> meetings.api.ts generateMeetingSummary(joinCode)
      -> POST /api/meet/[joinCode]/summary   (Next route.ts)
          -> meetingSummarization.service.ts summarizeMeetingForJoinCode
              -> summarizer.client.ts summarizeViaPythonSummarizer
                  -> POST http://127.0.0.1:8000/summarize (python api.py)
                      -> _stub_summary + (optional) Stage-A extractive digest
                  <- { summary, model, meta }
          <- { summary, model?, meta? }
  <- MeetingSummaryResult (client-side)
```

## Boundary Contracts (Do Not Change)

### 1) Next -> Python contract (via `summarizer.client.ts`)

**Request body**

```json
{
  "meetingId": "optional",
  "transcript": "string",
  "options": { "format": "bullets|paragraph", "maxTokens": 220 }
}
```

**Response body**

```json
{
  "summary": "string",
  "model": { "name": "string", "version": "string" },
  "meta": { "latencyMs": 0, "charsIn": 12345 }
}
```

### 2) Browser -> Next summary route (via `meetings.api.ts`)

**Request**
`POST /api/meet/:joinCode/summary` with JSON `{ format, maxTokens }`

**Response**
Same shape as Python response (forwarded back).

## Data Model (Prisma)

### Meeting

- `transcript: String?` — flattened transcript text
- `hasTranscript: Boolean`
- `transcriptCreatedAt: DateTime?`
- plus meeting metadata (`joinCode`, status, etc.)

### MeetingSegment

- `meetingId`
- `startMs`, `endMs`
- `speakerIndex` (numeric)
- `text`

### MeetingSpeaker

- `meetingId`
- `speakerIndex`
- `label` (e.g., “Speaker 1”)

> Current summarization uses `Meeting.transcript` (flattened blob), even though segments exist.

# File-by-file: What Each One Does

## Frontend client: `meetings.api.ts`

**Purpose:** Convenience wrappers that call Next routes.

Relevant functions:

- `generateMeetingSummary(joinCode)`
  - calls: `POST /api/meet/{joinCode}/summary`
  - sends: `{ format: "bullets", maxTokens: 220 }`
  - returns: `{ summary, actionItems: [], generatedAt, provider }`
  - `provider` is guessed based on `data.model?.name === "stub"`

Other related funcs (transcription pipeline support):

- `transcribeMeeting`, `fetchMeetingTranscript`, `saveMeetingTranscript`, etc.

---

## Next route: `route.ts` (summary API)

**Purpose:** Auth + HTTP boundary for summary.

- Reads session cookie (`SESSION_COOKIE`)
- Calls `AuthService.getMeFromSessionId`
- Parses request body `{ format, maxTokens }`
- Calls:
  - `summarizeMeetingForJoinCode({ joinCode, userId, options })`

Error mapping:

- `401` if missing/invalid session
- `403` ForbiddenError
- `404` NotFoundError
- `409` TranscriptMissingError (JSON `{ error }`)
- `500` generic

---

## Service: `meetingSummarization.service.ts`

**Purpose:** Business rules for “summarize a meeting”.

- Loads meeting by joinCode (`MeetingRepo.findMeetingByJoinCode`)
- Checks membership (`MeetingRepo.isProjectMember`)
- Pulls transcript from `meeting.transcript`
- Throws `TranscriptMissingError` if empty
- Calls Python client: `summarizeViaPythonSummarizer({ meetingId, transcript, options })`

Important notes:

- **No summary persistence** in DB (currently returned only).
- **Does not use MeetingSegments** yet (even though they exist).

---

## Client: `summarizer.client.ts`

**Purpose:** The only TS module that knows the Python service exists.

- Uses `process.env.SUMMARIZER_API_BASE_URL`
- Calls:
  - `POST {BASE_URL}/summarize`

- Throws `SummarizerUnavailableError` if base URL missing or non-200 response.

This file is the “bridge” boundary; keep it stable.

---

## Repo: `meeting.repo.ts`

**Purpose:** Prisma access for meetings.

Provides:

- `isProjectMember(projectId, userId)`
- `createMeeting(...)`
- `findMeetingByJoinCode(joinCode)`
- `findMeetingWithMembersByJoinCode(joinCode)`
- `updateMeetingStatus(...)`
- `setMeetingHasRecording(...)`
- `updateMeetingTranscript(meetingId, transcript)`
  - sets `hasTranscript=true` and `transcriptCreatedAt`

---

## Repo: `meetingSegment.repo.ts`

**Purpose:** Prisma access for segments.

Provides:

- `deleteSegmentsForMeeting(meetingId)`
- `bulkInsertSegments(meetingId, segments[])`
- `listForMeeting(meetingId)` (ordered)
- `getSegmentsForMeeting(meetingId)` (same as list; duplicate API)

Segments are persisted during transcription.

---

## Repo: `meetingSpeaker.repo.ts`

**Purpose:** Prisma access for speaker labels.

Provides:

- `getSpeakersForMeeting(meetingId)` (ordered)
- `replaceSpeakersForMeeting(meetingId, speakers[])`
  - deletes then inserts

---

## Service: `meetingTranscription.service.ts`

**Purpose:** Create transcript + diarized segments from latest recording.

Key flow (transcribe):

- Loads meeting by joinCode
- Checks membership
- Finds latest recording in `MeetingRecordingRepo` (not included here but referenced)
- Downloads recording from S3 (`S3_BUCKET`, `S3_REGION`, keys)
- Calls OpenAI transcription:
  - model: `gpt-4o-transcribe-diarize`
  - response format: `diarized_json`

- Converts diarized speaker IDs to numeric indices
- Builds `segments[]` with `{ startMs, endMs, speakerIndex, text }`
- Builds `fullTranscript`:
  - if diarized segments exist: join segment text with spaces
  - else fallback to plain `result.text`

- Persists:
  - `MeetingSegmentRepo.deleteSegmentsForMeeting`
  - `MeetingSegmentRepo.bulkInsertSegments`
  - `MeetingSpeakerRepo.replaceSpeakersForMeeting` (default `Speaker N`)
  - `MeetingRepo.updateMeetingTranscript`

Also provides:

- `getTranscriptForJoinCode` returning segments + speaker labels with fallback labels.

## Python FastAPI: `python-summarizer/api.py`

**Purpose:** Provide local summarization endpoint at `/summarize`.

Routes:

- `GET /health`
- `POST /summarize`

### Current summarization behavior

- Still stub-based:
  - `summary = _stub_summary(req.transcript, req.options.format)`

- Returns:
  - `model: { name: "stub", version: "0.0.1" }`
  - `meta: { latencyMs, charsIn }`

### Stage A extractive (already implemented)

Inside `_stub_summary`:

- Cleans transcript for Stage A:
  - removes “`[mm:ss] X:`” style prefixes per line
  - normalizes whitespace

- Uses threshold:
  - `_EXTRACTIVE_THRESHOLD_CHARS = 6000`
  - only runs extractive if transcript > threshold

- Extracts ~12 key sentences:
  - heuristic scoring (keywords like decision/action/blocker)
  - de-prioritizes obvious counting (“one, two, three”)

- Appends digest into the **same `summary` string** when used.

> This is intentionally “visible output” for testing now, without changing JSON contract.

# Current Limitations / Known Gaps

1. **No summary persistence**

- Summary is returned but not stored in DB (no `MeetingSummary` table).

2. **Stage B (abstractive) is stubbed**

- Only Stage A extractive is real logic today.
- Later, the replace point is the line in `/summarize` that calls `_stub_summary(...)`.

3. **Extractive threshold is heuristic**

- 6000 chars is a starting value; may need tuning per your transcript sizes.
